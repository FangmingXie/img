# Cell segmentation — findings (cell_seg)

Date: 2026-06-26. Source image: `links/cell_seg/cell_segmentation_Saumya.tif`.
Plans: `plan/cell_seg/{00,01,02}.*.md`. Scripts: `scripts/cell_seg/`.
Outputs (gitignored): `local_data/res/cell_seg/`, `local_data/fig/cell_seg/`.

## 1. The data

- Deconvolved Nikon AX confocal stack, ImageJ hyperstack, **ZCYX = 39×4×262×266, uint16**.
- Originally 1024×1024 in the source `.nd2`; the saved 262×266 field is at native
  resolution (a crop, not a downsample — see voxel size below).
- **Voxel size (confirmed by user, 2026-06-26):** XY = **57.5 nm/px** (matches the
  embedded TIFF resolution tag), Z = **160 nm/step**. → anisotropy (Z:XY) = **2.78**,
  voxel volume = **5.29×10⁻⁴ µm³**. Volumes below are now in real µm³.
- *Drosophila* sample (`slp1`, `beat-IIa`, "lobe1"). 4 channels:
  - **C0 = DAPI / nuclei** (dense, uniform; confirmed via `01.preview_channels.py`) — segmentation target.
  - C1 diffuse/dense; C2, C3 punctate sparse (specific markers).
- Nuclei are **large and densely packed** (~30–40 px across in XY) with visible
  **sub-nuclear DAPI texture**, and signal **dims with depth**. These three facts
  drove every segmentation difficulty below.

## 2. Environment lessons (the expensive one)

- The project env `img` was built **entirely from conda-forge in a single solve**:
  `python 3.11 + numpy/scipy/scikit-image/tifffile/pandas/matplotlib + pytorch-gpu + cellpose`.
  Verified: torch 2.12.1 (cuda130), `cuda=True` on RTX 6000 Ada; cellpose 4.1.1.
- **Do NOT `pip install cellpose` into a conda env that already has conda torch.**
  pip pulled a newer torch (2.12.1, cu13 wheels) that overwrote the conda torch; its
  `optree` dep then failed at import with `GLIBCXX_3.4.31 not found` — the pip wheel
  loaded the **system** libstdc++ (too old) instead of the env's. The conda-forge
  build links against conda's modern libstdc++, so the single-solve approach just works.
  (Full write-up in `plan/cell_seg/00.package_install_plan.md`.)
- cellpose on conda-forge tops out at **4.1.1** (pip has 4.2.x). 4.1.1 is current
  Cellpose-SAM and fine here; not worth the pip breakage.

## 3. Cellpose 4 (SAM) API notes

- The old `models.Cellpose(model_type=...)` class is **gone**. Use
  `models.CellposeModel(gpu=True)` (default `pretrained_model='cpsam'`).
- `model.eval(...)` returns **3** values `(masks, flows, styles)` — no `diams`.
- Single-channel 3D input: pass `(Z,Y,X)` with `z_axis=0, channel_axis=None`; the old
  `channels=[...]` arg is obsolete for cpsam. `diameter=None` uses cpsam's ~30 px default.
- Modes: 2D+stitch = `do_3D=False, stitch_threshold>0`; true 3D = `do_3D=True, anisotropy=…`.

## 4. Methods tried and results

Same field, same DAPI channel, scored with the shared `03.measure_and_qc.py`
(per-cell CSV + count + QC overlays). Volumes now in **µm³** (voxel = 5.29×10⁻⁴ µm³).
3D runs use the **correct anisotropy 2.78**.

| Method | Script | Count | Median vol (µm³) | Time | Character |
|---|---|---:|---:|---:|---|
| Watershed (first pass) | 02b | 171 | — | <1 s | severe over-segmentation |
| Watershed (tuned) | 02b | **72** | 4.90 | <1 s | decent per-slice; still splits/merges |
| Cellpose 2D+stitch | 02 | **50** | **7.09** | 5.3 s | cleanest contours; loses deep nuclei |
| Cellpose 3D (`do_3D`) | 02c | **146** | 0.52 | 23.3 s | recovers depth but heavily over-fragments |

**Biological sanity check:** *Drosophila* nuclei are ~2–4 µm across → ~4–14 µm³.
Cellpose 2D's median **7.09 µm³** sits squarely in range; watershed's 4.90 µm³ is a
touch low (mild over-split); Cellpose 3D's **0.52 µm³** is an order of magnitude too
small — quantitative confirmation of its over-fragmentation.

### Qualitative
- **Watershed** over-segments because sub-nuclear DAPI texture creates multiple
  distance-transform peaks per nucleus. Fix that mattered most: **smooth the distance
  map before seeding** (`DIST_SMOOTH`) + larger `MIN_DISTANCE` (171 → 72). Still the
  classic ceiling: touching nuclei merge, big/irregular ones split.
- **Cellpose 2D+stitch** gives by far the most nucleus-like contours (single closed
  boundaries, no internal fragmentation) — the clear quality winner per slice. But it
  **drops nuclei in dim/blurry deep slices** (per-slice counts fall from ~15 to ~1 by
  the last z), and some masks merge neighbors (very large max volume).
- **Cellpose 3D** catches the deep nuclei the 2D path misses, but **over-fragments** —
  and this persists **even with the correct anisotropy (2.78)**: median volume stays
  ~0.5 µm³ and the count rose 130→146. So 3D fragmentation here is *not* an anisotropy
  artifact; cpsam's `do_3D` flow simply breaks up these textured/dim nuclei. Important
  negative result — getting the real Z-step did **not** rescue the 3D path.

## 5. Key challenges / open issues

1. **Anisotropy resolved** (Z:XY = 2.78) but it did **not** fix Cellpose 3D — `do_3D`
   over-fragments regardless. 2D+stitch remains the better path here.
2. **Depth-dependent dimming** — per-Z percentile normalization + `cellprob_threshold=-1.0`
   helped only marginally for 2D+stitch (45 → 50); deep slices still fade out. This,
   not anisotropy, is now the main limiter on 2D+stitch recall.
3. **No ground truth** — counts span 50 (2D) – 146 (3D). The µm³ sanity check favours
   Cellpose 2D, but a manual count on a few slices is still needed to pin recall.
4. **Native-resolution crop** — the saved field is a 262×266 ROI of the 1024×1024
   acquisition at 57.5 nm/px; full FOV would need re-export from the `.nd2`.

## 6. Recommendations / next steps

- ✅ **Voxel size obtained** (XY 57.5 nm, Z 160 nm) and wired into `02b`/`02c`
  (`anisotropy=2.78`) and `03` (µm³ volumes + µm centroids).
- **Adopt Cellpose 2D+stitch (`02`) as the default** — cleanest contours and the only
  method whose median volume (7.09 µm³) is biologically sensible. Deprioritize `02c`
  (3D over-fragments even with correct anisotropy).
- **Improve 2D+stitch recall in deep slices** — the remaining real problem. Try
  per-slice CLAHE / background flattening before cpsam, and sweep `STITCH_THRESHOLD`.
- **Validate with a manual count** on 2–3 slices to fix recall and choose final params.
- If full FOV is wanted, **re-export the whole 1024×1024 field from the `.nd2`** (the
  current file is a native-resolution crop).
- StarDist 2D (`versatile_fluo`) remains a sensible second opinion for dense round
  nuclei if Cellpose 2D's merges prove problematic.
