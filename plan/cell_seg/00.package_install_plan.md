# Plan 00 — Package installation for the `img` env

## Context

Prerequisite for the cell-segmentation work (Plans 01 & 02). All image-processing
packages go into the **single project env `img`** (currently bare py3.11) — no
borrowing from other conda envs. This plan is the one place package installation
is defined; the other plans just assume `img` is ready.

**Policy (from CLAUDE.md):** do not install without explicit user OK; snapshot the
current package list to `env/` before installing; try conda first, pip later.

System facts: GPU = 4× NVIDIA RTX 6000 Ada (48 GB, idle), so a CUDA build of
PyTorch is appropriate for Cellpose.

## Package groups

**Group A — scientific stack** (watershed baseline, channel preview, QC/measurement;
Plan 02 + the shared `03.measure_and_qc.py` + `01.preview_channels.py`):
`numpy  scipy  scikit-image  tifffile  pandas  matplotlib`

**Group B — Cellpose / deep learning** (Plan 01):
`pytorch-gpu` (CUDA build) + `cellpose` (Cellpose-SAM, v4) — both from conda-forge.

> **As-built note (2026-06-26).** The first attempt installed Group A via conda-forge,
> torch via conda-forge `pytorch-gpu`, then **cellpose via pip**. pip pulled a newer
> torch (2.12.1, cu13 wheels) that overwrote the conda torch; its `optree` dep then
> failed at import with `GLIBCXX_3.4.31 not found` because the pip wheel loaded the
> **system** libstdc++ (old) instead of the env's. Lesson: do **not** mix pip torch
> into a conda env. Fix = install everything (incl. cellpose, which is a pure-python
> noarch package on conda-forge ≤4.1.1) in a **single conda-forge solve**, so torch
> and its compiled deps link against conda's modern libstdc++. Steps below are the
> corrected, working procedure.

## Steps

### 0. Snapshot current env (before any change)
```
conda list -n img > env/img_conda_list_<date>.txt
conda run -n img pip freeze > env/img_pip_freeze_<date>.txt
```

### 1. Install everything in one conda-forge solve (conda-only, no pip torch)
Recreate the env so all packages are mutually consistent and conda-managed:
```
conda env remove -n img -y
conda create -n img -c conda-forge -y \
  python=3.11 \
  numpy scipy scikit-image tifffile pandas matplotlib \
  pytorch-gpu cellpose
```
(If the env has no pip overlay yet, a plain `conda install -n img -c conda-forge ...`
of the same list works too — recreation is only needed to undo a prior pip mess.)

### 2. Reinstall the editable project package
```
conda run -n img pip install -e .
```

### 3. Verify
```
conda run -n img python -c "import numpy,scipy,skimage,tifffile,pandas,matplotlib,torch,cellpose; \
from cellpose import models; \
print('torch', torch.__version__, 'cuda', torch.cuda.is_available()); \
m = models.CellposeModel(gpu=True); print('cellpose', cellpose.version, 'gpu', m.gpu)"
```
Expect `cuda True` and a working `CellposeModel` (first init downloads the ~1.15 GB
`cpsam` weights — one-time).

### 4. Re-snapshot after install
```
conda list -n img > env/img_conda_list_<date>_post_install.txt
conda run -n img pip freeze > env/img_pip_freeze_<date>_post_install.txt
```

## Notes / risks
- **Do not mix pip torch into the conda env** — it overwrites the conda torch and its
  compiled deps load the system libstdc++ (→ `GLIBCXX` errors). Keep torch + cellpose
  conda-forge-managed. (cellpose on conda-forge tops out at 4.1.1; pip has 4.2.x but
  is not worth the breakage here.)
- **GPU build is driver-compatible**: driver 580 (CUDA 13) runs the conda-forge
  `cuda130` torch build; newer driver than toolkit is fine.
- After this plan, *all* segmentation scripts run with `conda run -n img ...`.
